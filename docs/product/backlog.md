## Data model / schema

- [ ] **ID validator tightness in `apply_patch`.** Prod D1 has `cmt_rogelios-framing-001` and friends — the agent invented IDs because runtime validation is "non-empty string" ([`ids.ts`](../../packages/database/src/schema/ids.ts)) and `apply_patch` requires caller-supplied IDs for content-addressed hash determinism. Options: (a) tighten validator to `^prefix_[A-Za-z0-9_-]{21}$` (has a data-migration cost); (b) server-generate on `op: "create"` before hashing (changes the `apply_patch` contract); (c) describe-and-pray in tool description (cheap, leaky). Needs an ADR. Surfaced in [dogfood/testing-log.md](../dogfood/testing-log.md) (2026-04-20).
- [ ] **Activation `pricePortion` for unit-priced commitments.** Schema says `pricePortion: Money` — for unit-priced, is it the *expected* portion? Needs nailing down.
- [ ] **Fractional `estimatedUnits` × integer cents.** `Commitment.price.estimatedUnits` is `z.number().nonnegative()`, but the price-equals-activation-sum invariant compares integer cents. A fractional unit estimate (42.5 sqft × 7500¢ = 318750¢, fine; 42.5 × 7501¢ = 318792.5¢, rounds) can spuriously pass or fail the check depending on how the cents are reconstructed. Options: force-integer-cents invariant, tolerance-based comparison (±1¢), or tighten `estimatedUnits` to integer. Flagged in `packages/database/src/invariants/commitments.ts`.
- [ ] **Schedule event log: `DelayEvent` + activation closure.** Three-event model for an activation's lifecycle, M3+: (a) `NTPEvent` (already in SPEC §1 — proceed notice); (b) `DelayEvent` (new — FK'd to activation with `cause` ∈ {weather, site_block, owner_delay, sub_delay, other}, `startedOn`, `endedOn?`, `note?`; `finishBy = startBy + buildTime + sum(delay durations)`); (c) an activation-closure event marking actual-finish, enabling variance math. Folds in the `siteReady: boolean` concept dropped per [ADR 0007](../decisions/0007-ntp-derivation-from-current-activation.md) — a zero-duration `DelayEvent` with `cause: site_block` at NTP issue time supersedes the flag. Lands with M3+ schedule/pay-app work.
- [ ] **Commitment label disambiguation.** M3 uses bare `counterparty.name` as the display label in `cost_entry_form` (see [`resolveCommitmentLabel`](../../apps/mcp-server/src/tools/cost_entry_form.resolver.ts)). If a job ever has two commitments from the same counterparty, the label becomes ambiguous in the UI. Options: append a price summary ("Rogelio's Framing LLC — $8,500 lump"), append the primary scope name, or introduce an explicit `Commitment.label` field. Revisit when the first collision shows up in dogfood or at M4 scheduling work.
- [ ] **Direct-materials escape hatch.** Requiring every Cost to reference a Commitment is clean but annoying for petty cash / direct buys. Option: a per-job "self-commitment" that absorbs direct materials, auto-created.
- [ ] **Party FK wiring + `create_job` integrity sweep.** `Job.clientPartyId` is declared structurally in [`packages/database/src/schema/jobs.ts`](../../packages/database/src/schema/jobs.ts) but not wired into SQL FK constraints. `create_job` currently accepts caller-supplied `clientPartyId` as-is (no existence check). When the party FK lands, add an existence pre-check to `create_job` (and audit every other tool that takes a `PartyId`). Flagged in PR #7 / [`apps/mcp-server/src/tools/create_job.ts`](../../apps/mcp-server/src/tools/create_job.ts) module header.
- [ ] **% complete on pay apps.** Three plausible drivers: (a) operator-reported % per activation, (b) cost-to-committed ratio, (c) activation state (NTP'd=10%, started=50%, finished=100%). Likely (a) with (b) as an AI-suggested default. Needs a `ProgressReport` event? Or a field on Activation?
- [ ] **Billed vs. paid.** Nothing in the schema currently tracks billing/payment state. Add `PayApp` as a first-class event (numbered, approved, paid)? Add `Payment` as another append-only event referencing a Cost or PayApp line?
- [ ] **Retainage.** Field on Commitment, field on each pay app line, or per-job default?
- [ ] **Scope taxonomy.** CSI codes vs. custom lightweight list vs. free-form. Commercial CSI feels heavy for residential custom. Leaning: free-form `name` + optional `code`, with a curated seed list. *(Currently: free-form; see [SPEC.md §1](../../SPEC.md).)*
- [ ] **Realized vs. unrealized variance.** `committed − cost` is a single number that conflates "activation under-ran (genuine favorable variance)" with "activation hasn't started yet (just unrealized commitment)." Per-activation state (NTP'd / started / closed) is needed to distinguish — without it, every active job looks like it's running under budget. Surfaces directly in the M4 dashboard. Related to "% complete on pay apps" above. Surfaced in [dogfood/testing-log.md](../dogfood/testing-log.md) (2026-04-20).
- [ ] **Self-commitment rollup semantics.** Self-commitments (created by `record_direct_cost`) contribute to `committed` the same as sub commitments — but conceptually a self-commitment is "already paid, no future obligation," not really "committed." Direct costs are also variance-neutral by construction (`record_direct_cost` inflates committed by exactly the cost amount), so per-scope variance goes blind on them. Forks: add a `selfFunded: boolean` flag excluded from committed rollups, or keep uniform and teach dashboards to render differently. M4-relevant. Surfaced in [dogfood/testing-log.md](../dogfood/testing-log.md) (2026-04-20).
- [ ] **Project = property convention.** Agent named the dogfood project "Nick Richards Kitchen Remodel" — collapsing client + work-type — instead of [TOOLS.md §6](../../TOOLS.md)'s `project = property, job = work-type` convention. `create_project`'s tool description doesn't prescribe the convention; multi-job-per-property scenarios (a second reno on the same address) don't fit cleanly into the chosen grain. Options: (a) tighten `create_project` description (also part of the tool-description polish in Runtime/MCP), (b) add validation hinting, (c) leave open. Surfaced in [dogfood/testing-log.md](../dogfood/testing-log.md) (2026-04-20).
- [ ] **Holiday calendar in working-days math.** [`_working-days.ts`](../../apps/mcp-server/src/tools/_working-days.ts) hard-codes Mon–Fri with no holiday calendar — Memorial Day, July 4, Labor Day all count as working days. Systematic earlier-finish bias for US-based subs. Forks: hard-code US federal holidays, accept a per-job holiday list, or integrate a calendar library. Surfaced in [dogfood/testing-log.md](../dogfood/testing-log.md) (2026-04-20).
- [ ] **ScopeSpec vocabulary.** Currently closed (`materials`, `installNotes`, `planRef`, `optionRef`); Zod silently strips unknown keys. Forks: keep closed for consistency, open via `.passthrough()`, or add an explicit `extras: Record<string, unknown>` field. Bites the first time someone wants to attach scope-level data the schema doesn't anticipate (e.g. punch-list checkboxes, pre-existing-conditions photos). Surfaced in [dogfood/testing-log.md](../dogfood/testing-log.md) (2026-04-20).
- [ ] **Self-party / counterparty dedup.** No canonical `selfPartyId` on Job — every `record_direct_cost` in a fresh session can create duplicate "Max" parties. Candidates: a `Job.selfPartyId` field, or a dedup hint in `create_party`'s description. Surfaced in [dogfood/testing-log.md](../dogfood/testing-log.md) (2026-04-20).
- [ ] **Approvals / lien waivers schema.** [TOOLS.md §5.3](../../TOOLS.md) names `lien_waiver_tracker` as an M5+ MCP app, but no entity captures how lien waivers relate to commitments / activations / costs. Same gap for general approvals (NTP issuance approval, change-order approval). Needs schema design before the M5 UI lands. Surfaced in [dogfood/testing-log.md](../dogfood/testing-log.md) (2026-04-20).
- [ ] **Client-side ledger (upstream).** Current model captures only what we owe subs (downstream commitments + costs). Missing: what clients owe us — receivable commitments, pay-app-side events. "Eating a sub cost" vs. "passing through with markup" is currently unrepresentable. Two paths: add `direction: payable | receivable` to Commitment (minimal change, forced symmetry), or introduce a separate `ClientAgreement` / `Sale` entity with its own shape (pay schedules / milestones / retainage don't always map cleanly to activations). Strategic; pairs with M5 pay-app work. Surfaced in [dogfood/testing-log.md](../dogfood/testing-log.md) (2026-04-20).
- [ ] **Bids as first-class entity.** `Commitment` has `signedOn` but no "pending bid" state; pre-signed contract tracking happens entirely outside the model. Three forks: add a `Bid` entity (mirrors Commitment shape pre-signature), add `state: bid|signed|void` to Commitment, or keep "Commitment = signed contract" and use external tooling for bids. Surfaced in [dogfood/testing-log.md](../dogfood/testing-log.md) (2026-04-20).
- [ ] **Activation dependencies.** No `dependsOn` / `predecessors` field on Activation; gantt and critical-path math aren't derivable from the current model. Required for any schedule visualization beyond per-activation start/finish dates. Surfaced in [dogfood/testing-log.md](../dogfood/testing-log.md) (2026-04-20).
- [ ] **Scope templates / per-project reuse.** `scopes.jobId` is NOT NULL — every job rebuilds its tree from scratch, even when the work is structurally identical (e.g. the same kitchen-remodel layout repeated on a second property). Forks: templates-as-clones (`clone_scopes({fromJobId, toJobId})` or a `ScopeTemplate` entity), or promote Scope to project level (deeper — breaks the "scope belongs to a job" invariant). Surfaced in [dogfood/testing-log.md](../dogfood/testing-log.md) (2026-04-20).

## Patches / event sourcing

- [ ] **`parentPatchId` required for non-create edits.** Every change order is by definition derived from a prior patch — the field exists for provenance. Consider requiring it on any `apply_patch` whose `edits` include non-`create` ops, so the chain stays explicit. Cheap hygiene. Surfaced in [dogfood/testing-log.md](../dogfood/testing-log.md) (2026-04-20).
- [ ] **Authoring UX.** Does the operator batch edits into a patch consciously, or does the server auto-group edits within a session?
- [ ] **Concurrency.** Single-operator is fine for dogfood. Two-operator on one job eventually means two patches at once. Git-style branching? Linear chain with optimistic concurrency? Likely post-v1.
- [ ] **Patch granularity.** Should a patch span multiple jobs (project-level CO affecting two jobs)? Current schema says no.

## System topology

Vision-level questions that aren't data-model or runtime-mechanic questions — they shape *how* the system is structured across milestones. Surfaced together in the 2026-04-20 dogfood session ([dogfood/testing-log.md](../dogfood/testing-log.md)); the testing log notes they're "not independent decisions; candidate for an 'architecture notes' doc sketching the interactions."

- [ ] **Agent topology.** Single general-purpose agent (on-demand, broad context) vs. specialized skills (on-demand, narrow) vs. scheduled intelligence (cron-initiated, no user). Likely per-capability: e.g., "process bids from inbox" → scheduled; "draft a change-order response" → on-demand-general; "issue NTP" → scheduled-skill + operator-attestation MCP app.
- [ ] **Integration ownership.** Us-as-integrator (we host `send_lien_waiver` API + status webhooks — predictable, auditable) vs. Claude-as-orchestrator (we expose data + verbs; Claude stitches Gmail + DocuSign + our MCP — flexible, no API maintenance). MCP-first ethos leans orchestration, but audit-critical flows (payments, lien-waiver status) may need us-as-integrator for reliability/traceability.
- [ ] **External-party UX.** Subs and clients don't have Claude Desktop. Inbound rides on the channels they *do* have — email, SMS, web. Forks: build-custom-forms (lightweight web app + form builder), integrate-existing (Notion DBs, Google Forms, DocuSign), or email-only degenerate. Processed inbound → scheduled-skill pattern from the agent-topology question.
- [ ] **Operator UX split (MCP apps vs. web app).** MCP apps strong for attestation (the click *is* the product, per the M3 pattern); weak for scanning-across-rows (dashboards, bulk edits, filtering). Read-only web app paired with attesting MCP apps may be the right split. Existing specced apps (`job_dashboard` M4, `pay_app_preview` M5 in [TOOLS.md §5.3](../../TOOLS.md)) are all MCP apps — worth reconsidering whether dashboard shape fits the MCP-app form factor before M4 builds.
